# HW-1 Working Agreements

Нюансы, как мы договорились работать на HW-1. Каждая новая сессия должна это
прочитать. Когда HW-1 закроется — убрать ссылку из `CLAUDE.md`.

## Роль агента

Это не автодополнение, а AI SWE в режиме SDD. Основной продукт сессии —
**спецификация**, а не код. Код — производная от отшлифованной спеки.
См. `Материалы1.md` секции «AI Software Engineer» и «Spec-Driven Development».

## Стиль взаимодействия

- **Объяснять каждый шаг — что / почему / зачем.** Пользователь учится CI,
  coverage, SDD-процессу. Плотные объяснения важнее скорости. Не «сделал»,
  а «сделал X, потому что Y, иначе Z».
- **Не коммитить молча.** Перед каждым коммитом показать какие файлы в stage
  и что в них изменилось. Commit только после подтверждения либо после явного
  «коммить» в рамках текущей задачи.
- **При сомнениях — спрашивать, а не угадывать.** Особенно: git destructive
  операции, добавление зависимостей, изменения в CI, правки существующих
  миграций (запрещены по CLAUDE.md), трогание чужого hw-0-кода без явной
  необходимости.
- **Декомпозиция на шаги с TodoWrite.** Подготовительная фаза HW-1 — это
  8 шагов (см. `report.md`). Циклы по фичам — 6 шагов каждый (Brief → Brief review →
  Spec → Spec review → Plan → Plan review → Implement → Verify → Ship).
  TodoWrite обновляем в реальном времени.

## SDD pipeline — жёсткие правила

- **Статусы документов:** `draft → active`. В `plan.md` / `spec.md` / `brief.md`
  должен быть frontmatter с `status:`. Агент передаёт на следующий шаг только
  `active`-документ. Brief в `draft` не идёт в Spec, Spec в `draft` не идёт в Plan.
- **Ревью до 0 замечаний.** Промпты ревью — в `PROMPTS.md`. Критерий остановки
  цикла ревью — «0 замечаний, документ готов» от ревьюера. Если ревьюер выдаёт
  явно неадекватные замечания (бывает) — фиксируем это в `report.md` с комментарием,
  почему проигнорировали, и идём дальше. Типичное число итераций:
  Brief 2–3, Spec 3–5, Plan 1–2.
- **Линковка.** Каждая фича привязана к GitHub issue. В issue — ссылки на
  `brief.md`, `spec.md`, `plan.md` в репо. В каждом документе в frontmatter —
  `related_issue: #N`. PR закрывает issue через `Closes #N`.

## Стратегия сессий (из `session-strategies.md`)

- **Дефолт — уровень 3:** отдельная сессия на каждый артефакт. Brief в одной
  сессии, Spec в другой, Plan в третьей, Implement в четвёртой. Каждая стартует
  с чистого контекста.
- **F1 (Property CRUD) — уровень 4:** ревью с исправлением до 0 замечаний в той же
  сессии. F1 — эталон, на нём калибруем процесс, дисциплина выше.
- **Между сессиями передаём handover-текст** со ссылками на артефакты, todo и
  текущий статус. Не продолжаем длинную сессию, когда задача сменилась.
- **Watch context usage.** Если контекст подходит к 75% — завершить текущую
  подзадачу и передать в новую сессию. Признаки деградации: агент «забывает»
  ранние инструкции, повторяется, путает имена.

## Reference Implementation pattern

- **F1 = эталон CRUD.** Делается с повышенным вниманием, осознанным выбором
  паттернов (structure контроллера, error handling, scope к `Current.organization`,
  Pundit policy shape, request spec structure). Всё последующее (F2–F5) делается
  «по аналогии с F1».
- После F1 — обновить `CLAUDE.md` секцией **Reference implementations** с путями:

  ```text
  - CRUD controller: app/controllers/api/v1/properties_controller.rb
  - Pundit policy: app/policies/property_policy.rb
  - Request spec: spec/requests/api/v1/properties_spec.rb
  - Factory: spec/factories/properties.rb
  ```

- При генерации F2+ в промпте к агенту обязательно указывать эти пути:
  «Структура по аналогии с `app/controllers/api/v1/properties_controller.rb`».

## Coverage ratchet

- **Механизм:** `backend/spec/spec_helper.rb` содержит `minimum_coverage line: N`.
  После каждого merge в `hw-1` ветке смотрим фактический `Line Coverage: X%` в
  выводе rspec, поднимаем `N` до `floor(X) - 1` (маленький буфер против флейков).
- **История подъёма** фиксируется в `homeworks/hw-1/report.md` секция
  «Coverage ratchet».
- **К финальному PR** (`hw-1` → `main`) порог должен быть **80** — это требование
  сдачи. Если после 5 фич не дотягиваем — добавить тестов на существующий hw-0
  код (controllers, models).
- **Frontend** — аналогичный ratchet в `frontend/vitest.config.js`, стартует с 0.
  Если на HW-1 не успеем доставить реальные Vue-тесты — фронт остаётся с «placeholder
  coverage» и 0% порогом, финальная цель 80% переносится в HW-2.

## Git

- **Ветка hw-1.** Все коммиты HW-1 — в ветке `hw-1`, не в `main`. Финал —
  один PR `hw-1 → main`.
- **Fine-grained коммиты.** Каждый логический шаг = свой коммит. Не «всё одним
  коммитом в конце». Это даёт чистую историю и возможность bisect.
- **Commit messages:** английский, сабжект 50–72 символа, body поясняет «why».
  В конце — `Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>`.
- **Никаких `--force`, `reset --hard`, `rebase -i` без явного разрешения.**
  Даже на своей ветке.

## CI

- Все изменения проверяются локально (`bundle exec rspec`, `yarn test:coverage`,
  `yarn build`) **до** пуша.
- Badges обновляются только на push в `main` (соответствующий job имеет
  `if: github.ref == 'refs/heads/main'`). На ветке `hw-1` они будут «invalid» —
  это нормально, станут валидными после мержа.

## Документы после каждой фичи

По требованию CLAUDE.md «Documentation sync» — после merge каждой фичи обновить:

- `ai-docs/PLAN.md` — отметить `[x]`
- `ai-docs/SCHEMA.md` — если добавились модели/поля
- `ai-docs/DECISIONS.md` — если было нетривиальное архитектурное решение (DEC-NNN)
- `PROJECT.md` — если поменялось описание модулей
- `homeworks/hw-1/report.md` — строку фичи в сводной таблице + блок фичи
- `homeworks/hw-1/PROMPTS.md` — если по итогам ревью адаптировали промпт

Ни одна фича не считается сделанной, пока эти обновления не проведены.
