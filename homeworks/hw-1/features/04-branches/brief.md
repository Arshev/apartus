# Brief — Branches (self-referential tree)

## Контекст

Apartus — PMS для управления арендной недвижимостью. На момент старта
F4 в системе уже есть `Property` (F1), `Unit` (F2), и `Amenity/UnitAmenity`
(F3). Все они существуют в **плоской** структуре внутри организации:
менеджер видит один общий список объектов своей организации, один
общий каталог amenities, и так далее. Для небольшой компании с 5–10
объектами в одном городе этого достаточно.

Но целевой сегмент Apartus шире: **управляющие компании с
распределённой структурой**. Типичные сценарии:

- Сеть хостелов в разных городах. Каждый город — отдельное
  подразделение, в городе — свои хостелы, персонал, бюджет, отчётность.
- Региональный управляющий, отвечающий за несколько городов, которые
  группируются в «регион». Регион — ещё одно подразделение, над ним
  директор.
- Холдинг с вертикалью: головной офис → региональные центры → города
  → объекты. Четыре уровня иерархии в организации.

Все эти структуры моделируются **деревом подразделений** (branches) с
self-reference: у филиала есть `parent_branch_id`, а у корневых
филиалов родителя нет (`nil`). Это классическая **adjacency list** —
один столбец `parent_id` на записи, дерево строится рекурсивно.

F4 — **первая tree-форма в проекте**. В F1/F2/F3 отношения были
плоскими (belongs_to) или M:N (has_many :through). Дерево даёт новый
класс вопросов: что такое «корень», как считать ancestors/descendants,
что делать при удалении узла с детьми, как защитить от циклов
(«A — родитель B, B — родитель A»), как рендерить иерархию в API.

F4 намеренно **ограничен самой моделью Branch**: связь
`Branch ↔ Property` (какой branch владеет каким объектом) — отдельная
фича F5. Это позволяет отработать tree-паттерн в чистом виде, без
привязки к существующим ресурсам, и потом добавить связь в F5 минимальным
диффом.

**Branch — единица организационной структуры, а не контейнер объектов.**
Филиал имеет самостоятельный смысл: менеджер строит оргструктуру
компании (иерархию подразделений, кто кому подчинён) **до** того, как
начнёт привязывать к ней объекты, персонал, отчётность. Поэтому CRUD
филиалов имеет смысл даже без связи с Property — это «скелет»
компании, на который потом навешиваются ресурсы.

**Ожидаемый размер дерева — десятки узлов на организацию.** Точный
верхний предел (если таковой будет введён) — вопрос Q8. На HW-1
scale производительность queries не критична, но решение Spec по Q4
(форма API для чтения дерева) должно быть таким, чтобы `index`
эндпоинт не вводил N+1: либо один SQL запрос с плоским массивом,
либо явная eager loading стратегия. Это не отдельный NFR, а общее
ограничение на качество реализации.

**HW-1 не включает frontend.** API F4 — предусловие для UI Phase 2.3
и для scope membership Phase 1.3. В HW-1 функциональность F4 проверяется
через request-спеки (curl/Postman — при желании), реальное
пользовательское использование — после появления UI.

## Проблема

Менеджер управляющей компании с несколькими филиалами **не может
отразить свою организационную структуру в Apartus**. Из-за этого:

- Невозможно разделить видимость данных между филиалами на уровне
  структуры: «московский менеджер должен видеть только московские
  объекты, и не петербуржские». Сейчас все объекты организации видны
  всем сотрудникам с соответствующими permissions. Это будет решаться
  в Phase 1.3 (scope membership по branches), но предусловие — сама
  модель Branch.
- Невозможно сделать отчётность по подразделениям: «сколько
  реализовано в московском регионе за месяц», «какая загрузка в
  хостелах Санкт-Петербурга». Аналитика (Phase 6) требует агрегации
  по иерархии, и без самого дерева она невозможна.
- Невозможно делегировать административные функции региональному
  управляющему: «ты отвечаешь за всё, что под твоим региональным
  филиалом». Это паттерн Phase 7 (Owners + hierarchical permissions),
  но снова предусловие — Branch tree.
- Онбординг управляющей компании с распределённой структурой
  останавливается на первом шаге: «я не могу даже завести список своих
  подразделений, не говоря уже о привязке к ним объектов и людей».

## Стейкхолдеры

- **Первичный:** менеджер управляющей компании (роль `manager` или
  `owner` в Membership), который строит структуру филиалов своей
  организации и редактирует её по мере роста/реорганизации компании.
- **Вторичный:** разработчик Apartus — получает первый tree-pattern
  в проекте и наблюдения по SDD для форм, где новый тип связи (self-ref)
  меняет форму задачи.

**Не стейкхолдер F4:** сотрудник филиала (в F4 нет UI-фильтрации
«только мои филиалы»), региональный управляющий (scope membership по
branches — Phase 1.3, не HW-1), analytics (Phase 6). Все они появятся
в будущих фичах поверх Branch модели.

## Желаемый результат

После выполнения F4 менеджер с правом управления филиалами,
авторизованный в своей организации, может:

1. Посмотреть список всех филиалов своей организации.
2. Открыть детали конкретного филиала, включая его родительский
   филиал (если есть) и информацию о том, где в дереве находится этот
   филиал (корень / не корень).
3. Завести новый филиал:
   - корневой (без родителя);
   - дочерний (с указанием `parent_branch_id`).
4. Отредактировать филиал: изменить имя или родителя (включая
   возможность сделать филиал корневым). Что считать допустимым
   перемещением — вопрос Spec (Q2).
5. Удалить филиал. Поведение при удалении узла с детьми — вопрос для
   Spec (Q3).
6. Получить через API список филиалов своей организации в формате,
   пригодном для клиентского кода. Точная форма ответа (плоский
   список, вложенное дерево или оба) — вопрос Spec (Q4).

Все операции ограничены контекстом организации: филиалы другой
организации не видны, не редактируются, не могут быть выбраны в
качестве родителя, возвращают 404 без раскрытия существования.

### Критический инвариант безопасности F4

**Дерево филиалов одной организации никогда не пересекается с другой.**
Невозможно создать или отредактировать `Branch` так, чтобы
`parent_branch_id` указывал на филиал другой организации. Это
**блокирующее** требование: нарушение квалифицируется как security
bug, не как регрессия фичи. Spec обязан явно покрыть этот инвариант
тестом, а не полагаться на «общую логику scope'ов».

Это прямой аналог F2 §3.4.6 (cross-property isolation within same org)
и F3 §3.5.9 (cross-org amenity attach protection), но для tree-формы
риск выше: `parent_branch_id` — явный пользовательский input
(независимо от конкретной формы API, которую выберет Spec), и без
проверки он напрямую становится указателем в чужое поддерево.

После реализации:

- `ai-docs/PLAN.md` пункты `1.4.1` (модель Branch), `1.4.2` (иерархия
  parent/children), `1.4.4` (CRUD API) отмечены как выполненные.
  Пункт `1.4.3` (scope membership) **не** покрывается F4 — это Phase
  1.3/Phase 7.
- `ai-docs/DECISIONS.md` — новая запись DEC-NNN с обоснованием выбора
  представления дерева (per Issue #4 pipeline).
- `ai-docs/SCHEMA.md` — блок `Branch`.
- Coverage ratchet поднят.
- Reference pattern tree зафиксирован; наблюдения в `report.md`.

## Взаимодействие с F5 (Property ↔ Branch link)

F4 не реализует связь Branch ↔ Property — это явно scope F5. Но F4
накладывает на F5 один инвариант: **модель Branch и её таблица должны
быть такими, чтобы F5 мог добавить связь `Property belongs_to :branch,
optional: true` одной миграцией без изменения схемы branches**. То
есть:

- F4 не вводит на стороне `Branch` ничего, что зависит от
  существования `branches.property_id` или аналога.
- F4 не закладывает счётчики «сколько объектов в этом филиале» — это
  будет в F5 через counter cache или на лету.
- Удаление Branch в F4 не должно молча оставлять болтающиеся
  `property.branch_id` (потому что в F4 этого FK ещё нет, но в F5
  появится). Поведение `Property.branch_id` при удалении Branch
  решается в F5 (`nullify`/`restrict`/`cascade`), F4 это не
  предопределяет.

Этот инвариант — основа разделения F4 и F5 на две отдельные фичи. Его
нарушение в F4 потребует переделки модели Branch в F5, и фичи
склеятся обратно.

## Что НЕ входит (фиксируется явно, чтобы Spec не расширял scope)

- **Связь `Branch ↔ Property`.** Это явно F5 scope, не F4. Property
  остаются плоскими внутри организации до F5.
- **Scope membership по branches** (кто какие филиалы видит). Это
  Phase 1.3, не F4, не HW-1.
- **Scope permissions по branches** (региональный управляющий). Phase
  7, не F4.
- **Агрегация/отчёты по иерархии.** Phase 6.
- **Timezone, адрес, контактные данные филиала** — F4 минималистичен,
  только `name` и `parent_branch_id`. Доп. поля могут быть добавлены
  позже без breaking changes. (Конкретный минимальный набор полей —
  решение Spec, см. Q1.)
- **Мульти-root отдельная концепция.** F4 поддерживает множественные
  корневые филиалы в одной организации естественным образом
  (несколько записей с `parent_branch_id: nil`), но не вводит отдельную
  сущность «root» или «default root».
- **Переупорядочивание детей (ordering within siblings).** Список
  детей одного родителя возвращается в детерминированном порядке
  (Spec решит — по `id` или `name`), но пользователь не управляет
  порядком.
- **Drag-and-drop reorder в UI.** Вне HW-1 (нет frontend в HW-1).
- **Историчность дерева / audit log перемещений.**
- **Soft delete.** Branch удаляется физически (как и остальные
  ресурсы в проекте).
- **Bulk-операции** (перенести всех детей одного branch под другой
  одним запросом).
- **Frontend.**
- Любые изменения моделей Property, Unit, Amenity, UnitAmenity,
  Organization, Role, Membership, **кроме** добавления `has_many
  :branches, dependent: :destroy` на Organization.

## Открытые вопросы (для Spec-сессии)

Brief фиксирует контуры, Spec должен принять архитектурные решения:

**Q1. Зафиксировать в DEC-NNN выбор adjacency list.**
Это не открытая развилка: closure table добавляет ту же сложность
без выгоды на ожидаемом размере деревьев (десятки узлов на
организацию), а сторонние gem'ы (`ancestry`, `closure_tree`) запрещены
CLAUDE.md без явного согласования. Adjacency list — стандарт Rails:
один столбец `parent_branch_id` на записи, рекурсивные запросы через
PostgreSQL `WITH RECURSIVE` или Ruby-обход. Обязательный шаг per
Issue #4 pipeline — зафиксировать это в `ai-docs/DECISIONS.md` с
явным отказом от closure table и gem'ов.

**Q2. Защита от циклов.**
Если разрешено перемещение узла (`parent_branch_id` меняется через
PATCH), появляется риск цикла: `A.parent = B`, затем попытка
`B.parent = A`. Или хуже — `A.parent = A` (self-reference). Spec
должен явно:

- Запретить self-reference (`id == parent_branch_id`) — валидация
  модели, 422.
- Запретить создание цикла при update (ставить родителем своего
  потомка) — валидация или service-слой, 422.

Оба случая должны быть покрыты тестами. Первый — тривиален, второй —
требует обхода дерева перед сохранением.

**Q3. Поведение при удалении филиала с детьми.**
Варианты:

- **(а) Restrict** — запретить удаление узла, пока у него есть дети
  (409 Conflict, как F3 amenity-in-use). Пользователь должен сначала
  переместить или удалить детей.
- **(б) Cascade** — удалить узел и всё его поддерево (`dependent:
  :destroy` + рекурсивный обход). Риск: ошибочное удаление большого
  поддерева.
- **(в) Nullify / re-parent to grandparent** — дети перемещаются под
  родителя удаляемого узла; если удаляется корень — дети становятся
  корнями. Никаких данных не теряется.

Каждый вариант защищает разные пользовательские сценарии. Для HW-1
лучше всего **(а)** — явность и предсказуемость, согласовано с F3
restrict pattern. Spec должен это зафиксировать.

**Q4. Форма API для чтения дерева.**
Варианты:

- **(а) Плоский список** — `GET /branches` возвращает массив всех
  филиалов с `parent_branch_id`. Клиент строит дерево сам. Минимум
  работы сервера, максимум гибкости.
- **(б) Вложенное дерево** — рекурсивный JSON с
  `children: [...]` в каждом узле. Клиент сразу получает готовое
  представление. Тяжелее сериализация, неудобно для больших деревьев.
- **(в) И то, и другое** — два эндпоинта.

Рекомендация: **(а)** — минимальный подход, клиент может
использовать плоский массив как угодно. Вложенное представление —
оптимизация, не нужная на HW-1 scale.

**Q5. Минимальный набор полей Branch.**
Кандидаты:

- `name` (обязательно) — как F1/F3.
- `parent_branch_id` (nullable) — self-reference на `branches`.
- `organization_id` (обязательно) — per-org scoping.

Дополнительные поля, которые могут быть полезны, но **не** добавляются
в F4 без явного обоснования:

- `code` / `slug` — короткий идентификатор для URL и отчётов.
- `timezone` — часовой пояс филиала для reservation / финансов.
- `address` — адрес центра филиала.
- `description` — свободный текст.

Spec должен решить, добавлять ли `code`/`slug`/`timezone`/`address`
(и для каждого добавленного поля — правила нормализации, валидации,
уникальности), или оставить только name. F3 продемонстрировал, что
минималистический набор полей не блокирует будущие расширения
(новые nullable-колонки добавляются миграцией без breaking changes),
но это рекомендация по подходу, а не предопределённый ответ. Реальное
решение — за Spec, исходя из конкретных use case.

**Q6. Уникальность имени филиала.**
Варианты:

- **(а)** Уникально в рамках всей организации (никакие два филиала
  организации не могут иметь одно и то же имя).
- **(б)** Уникально в рамках одного родителя (в одном «Москва» может
  быть филиал «Центр» и филиал «Север», но не два «Центр»). Два
  разных «Москва.Центр» — ОК.
- **(в)** Нет уникальности вообще — менеджер сам следит.

Для tree-структур обычно используется **(б)** — «путь» в дереве
должен быть уникальным. Spec должен выбрать и обосновать.

**Q7. Permissions.**
Три варианта:

- **(а)** Отдельный код `branches.manage` (и `branches.view`) в
  `Permissions::ALL_PERMISSIONS`, как F3 amenities. Гранулярность,
  но ещё пара кодов в системе.
- **(б)** Покрыть существующим `organizations.manage` — branch это
  часть структуры организации, менеджер уровня organizations.manage
  и так управляет основами. Меньше кодов.
- **(в)** Без permission-кода вовсе, через явную проверку
  `role_enum == :owner` (как для удаления самой организации). Branch —
  редкая admin-операция уровня owner. Минимум гранулярности.

Spec должен выбрать и обосновать, сверившись с
`Permissions::ALL_PERMISSIONS` и существующей практикой ролей.

**Q8. Глубина дерева.**
Вводить ли верхнюю границу глубины (например, «не больше 5 уровней»)?
Варианты:

- **(а)** Ввести разумный лимит (5–10 уровней) с валидацией на
  create/update. Защита от опечаток и абсурдных деревьев.
- **(б)** Явно зафиксировать отсутствие лимита. Меньше кода, нет
  магического числа.

Spec должен явно решить, исходя из соображений безопасности и UX.
