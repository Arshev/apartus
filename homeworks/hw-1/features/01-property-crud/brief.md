# Brief — Property CRUD

## Контекст

Apartus — PMS для управления арендной недвижимостью. На текущий момент (конец
Phase 1) в системе есть аутентификация, мультитенантность (Organization +
Membership), роли и разрешения, но **нет ни одного бизнес-объекта предметной
области**. Пользователь, зайдя в организацию, видит пустой интерфейс — нечего
показать, нечего забронировать, не на чём считать финансы.

Все последующие модули PMS (Units, Reservations, Pricing, Payments, Owners,
Channel Manager) зависят от наличия **Property** — объекта недвижимости,
которым управляет организация. Без Property система функционально пуста.

## Проблема

Менеджер арендного бизнеса, который зарегистрировался в Apartus и создал
организацию, **не может завести в систему ни одного объекта недвижимости**.
Из-за этого:

- Невозможно начать пользоваться продуктом — онбординг обрывается сразу после
  создания организации.
- Невозможно продолжать разработку зависимых модулей (Units, бронирования,
  цены) — у них нет родительской сущности, к которой они привязываются.
- Невозможно показать работающий end-to-end слайс заказчику/преподавателю
  HW-1.

## Стейкхолдеры

- **Первичный:** менеджер арендного бизнеса (роль `manager` или `owner` в
  Membership) — заводит объекты недвижимости своей организации и редактирует
  их по мере изменений.
- **Вторичный:** разработчик Apartus — получает эталонный CRUD-паттерн
  (контроллер, Pundit-политика, request spec, scoping к организации), на
  который опираются все последующие фичи (F2–F5 в HW-1 и далее Phase 2+).

## Желаемый результат

После выполнения фичи менеджер, авторизованный в организации, может:

1. Посмотреть список всех объектов недвижимости **своей** организации.
2. Открыть детали конкретного объекта.
3. Завести новый объект, указав базовый набор атрибутов: название, адрес,
   тип (квартира, отель, дом, хостел), описание.
4. Изменить атрибуты существующего объекта.
5. Удалить объект, который больше не нужен.

Объекты одной организации **никогда не видны и не доступны** пользователям
другой организации — ни на чтение, ни на запись.

Пользователь с ограниченной ролью (обычный член организации без прав на
управление объектами) может только читать список, но не создавать, изменять
или удалять объекты.

## Критерии успеха (что считаем «фича работает»)

- Менеджер проходит полный цикл «создал → нашёл в списке → открыл → изменил →
  удалил» без ошибок на валидных входных данных.
- Попытка получить доступ к объекту чужой организации не раскрывает факт его
  существования.
- Пользователь без прав на управление объектами может читать список, но
  получает отказ при попытке создать/изменить/удалить.
- Поведение фичи зафиксировано автоматическими тестами, регрессия блокируется
  CI и порогом coverage ratchet.
- Модель Property готова служить родительской сущностью для последующих фич
  HW-1 (Units, Amenities, Property↔Branch).

## Ограничения и явные НЕ-цели

**НЕ входит в F1:**

- Связка `Property ↔ Branch` (`branch_id`) — отдельная фича F5.
- Модель `Unit` и nested-ресурс `properties/:id/units` — фича F2.
- `Amenity`, `UnitAmenity` — фича F3.
- Загрузка фото (Active Storage) — отложено на Phase 2.1.4.
- Frontend (Vue-страницы, формы) — вне HW-1 скоупа бекенд-эталона.
- Поиск, сортировка, пагинация за пределами простого `index` — отдельные
  задачи, если потребуется.
- Soft delete — по `ai-docs/SCHEMA.md § Conventions` не используется, если
  не принято отдельное DEC.

**Технические ограничения (из CLAUDE.md, обязательны, не обсуждаются):**

- Строгая изоляция данных между организациями — реализовать любым способом,
  не нарушая этот инвариант.
- Аутентификация и авторизация поверх существующих механизмов проекта
  (Rails 8 built-in auth, Pundit) — не вводить параллельных.
- Никаких новых gems/npm-пакетов без явного разрешения.
- Существующие миграции не изменяются — только новые.

## Почему сейчас

1. F1 — первая фича HW-1 и одновременно стартовая точка Phase 2 (Properties
   & Units) в `ai-docs/PLAN.md`. Без неё остальные фичи HW-1 не имеют базы.
2. F1 — **reference implementation**: по её паттернам будут делаться F2–F5.
   Важно пройти её с повышенной дисциплиной (session level 4, ревью до 0
   замечаний), чтобы зафиксировать паттерн один раз и переиспользовать.
3. Сводная сдача HW-1 требует работающего end-to-end цикла
   Brief → Spec → Plan → Implement с реальным кодом в репо — Property CRUD
   минимально достаточен как предметный кейс.
