# Brief — Property ↔ Branch link

## Контекст

Apartus — PMS для управления арендной недвижимостью. На момент старта
F5 в системе уже есть `Property` (F1), `Unit` (F2), `Amenity/UnitAmenity`
(F3), и `Branch` (F4). F4 отдельно подчёркивал, что связь
`Property ↔ Branch` **отложена до F5**: это сознательное разделение
на две фичи, чтобы отработать tree-паттерн в чистом виде и потом
добавить связь минимальным диффом.

F5 замыкает HW-1 и делает то, чего F4 намеренно избежал: **соединяет
Property с Branch**. После F5 каждый объект недвижимости может быть
привязан к конкретному филиалу организации, и менеджер может:

- посмотреть, какие объекты принадлежат какому филиалу;
- перевести объект из одного филиала в другой (реорганизация);
- оставить объект без филиала (свежий объект, ещё не распределённый).

С точки зрения доменной модели это **many-to-one**: много Property
могут принадлежать одному Branch, один Property принадлежит не более
чем одному Branch. Обратная сторона — Branch.has_many :properties —
становится технической, но **не** порождает новых endpoints в F5
(фильтрация объектов по филиалу — Phase 2.2.5, не HW-1).

F5 — **горизонтальная фича**: она не вводит новой сущности, а
изменяет существующие контракты F1 и F4. В отличие от вертикальных
F1–F4, где каждая фича вводила свою таблицу/модель/контроллер/policy,
F5 меняет то, что уже было: добавляет поле на `properties`,
расширяет `permit` в `PropertiesController`, модифицирует JSON-ответ
Property, изменяет `before_destroy` callback в модели Branch. Это
единственная такая фича в HW-1.

## Проблема

Менеджер управляющей компании с несколькими филиалами **не может
указать, к какому филиалу относится конкретный объект недвижимости**.
Из-за этого:

- Структура «филиал → объекты → юниты» не выражена: Property
  существуют плоско внутри организации, Branch существуют отдельно
  (F4), но связь между ними — пользовательский контракт, который
  приходится держать «в голове» или в сторонней таблице.
- Невозможна аналитика по филиалам: «загрузка объектов московского
  филиала», «доход регионального офиса». Phase 6 агрегация требует
  `Property.branch_id`.
- Невозможна фильтрация объектов: «показать все объекты в филиале
  HQ». Phase 2.2.5, не HW-1, но без `branch_id` это блокируется.
- Невозможно делегировать управление: региональный управляющий не
  может получить scope «только мои объекты», пока объекты не связаны
  со структурой. Phase 1.3/Phase 7.
- Онбординг УК завершается на неполной модели: «я завёл филиалы,
  завёл объекты — но они не связаны, всё это независимые списки».

F5 не решает Phase 6/2.2.5/1.3 — она **разблокирует** их, добавив
недостающее поле.

## Стейкхолдеры

- **Первичный:** менеджер управляющей компании (роль `manager` или
  `owner`) — привязывает свои Property к своим Branch, перемещает
  объекты между филиалами при реорганизации.
- **Вторичный:** разработчик Apartus — получает опыт «горизонтальной»
  фичи, которая затрагивает несколько существующих контрактов без
  введения новой сущности. Наблюдения по SDD фиксируются в
  `homeworks/hw-1/report.md` и важны для финальных выводов по
  процессу HW-1.

## Желаемый результат

После выполнения F5 менеджер с правом управления объектами,
авторизованный в своей организации, может:

1. При создании объекта (`POST /api/v1/properties`) указать
   `branch_id` — объект сразу привязан к филиалу. Или не указать —
   объект создаётся без филиала (`branch_id: null`).
2. Переместить существующий объект в другой филиал через `PATCH
   /api/v1/properties/:id {branch_id: N}`. Или отвязать, установив
   `branch_id: null`.
3. При чтении `GET /api/v1/properties` и `GET /api/v1/properties/:id`
   увидеть `branch_id` в JSON-ответе. Null для непривязанных
   объектов, integer для привязанных.
4. Попытка привязать объект к филиалу **другой организации** —
   отказ с кодом, согласованным с F4 security pattern (422 с
   не-раскрывающим сообщением, не 404). Детали — в Spec, но форма
   ошибки наследуется от F4.
5. При попытке удалить филиал, к которому привязаны Property —
   поведение, согласованное с Spec (Q1 ниже).

Все операции ограничены контекстом организации.

**Critical invariant: nullable `branch_id`.** Связь Property → Branch
— **опциональная**. Property может существовать без Branch (свежий
объект, ещё не распределённый по структуре). Это **осознанное**
решение: F4 Branch остаётся опциональной надстройкой над F1 Property,
не обязательной. Организация, которая не пользуется branches (одна
компания без внутренней структуры), продолжает иметь все Property
с `branch_id: null` — и F1 CRUD работает как раньше. Альтернатива
(обязательная связь) сломала бы F1 как reference implementation и
потребовала бы data migration.

После реализации:

- `ai-docs/PLAN.md` — пункт `1.4.5` Frontend остаётся `[ ]` (F5 не
  делает frontend), но F5 добавляет новые пункты, если нужны, или
  апдейтит формулировки существующих.
- `ai-docs/DECISIONS.md` — **не обязательно** новая DEC, но если
  выбор поведения `on_delete` для `Property.branch_id` нетривиален
  (см. Q1), добавить DEC-015.
- `ai-docs/SCHEMA.md` — блок Property обновлён с полем `branch_id`,
  блок Branch — упомянут `has_many :properties` в Associations.
- **Coverage ratchet: финальная цель HW-1 — `minimum_coverage line:
  80`** (per Issue #5 pipeline). Текущее после F4 — 72.38%, до
  цели нужно +7.62pp.
  Ожидание от F5 implementation: ~3–5pp (небольшая фича,
  ~20–30 новых тестов на ~80 строк production). Ожидаемое после
  F5 — 75–77%, т.е. всё ещё недостаточно.
  **Процедура:**
  1. После F5 implementation измерить actual coverage.
  2. Если `actual >= 80` — bump ratchet до `floor(actual) - 1`,
     HW-1 закрыт.
  3. Если `actual < 79` — добавить отдельный step в Plan «add
     tests for hw-0 code» (приоритет: controllers > models).
     Конкретные hw-0 файлы, требующие тестов, определяются после
     измерения — там, где coverage самый низкий. Возможные
     кандидаты: `auth/sessions_controller`, `auth/registrations_controller`,
     `members_controller`, `roles_controller`. Объём —
     ровно столько, чтобы поднять до 80%, не больше.
  4. Если `79 <= actual < 80` — пограничный случай, добавить
     минимум тестов, чтобы точно перейти 80.
- `homeworks/hw-1/report.md` — финальная строка F5 + итоговые выводы
  по HW-1.

**Regression guard — блокирующее требование.** F5 — первая фича,
которая **напрямую изменяет** существующий код F1 (PropertiesController,
Property model, Property JSON-контракт) и F4 (Branch model
`before_destroy` callback). Все существующие 289+ тестов F1/F2/F3/F4
после F5 implementation должны оставаться зелёными вместе с новыми
F5 тестами. Любое падение существующего теста — блокирующий баг,
требующий либо отката F5 implementation, либо явной ретроспективной
правки соответствующего Spec (F1 AC9 — единственный ожидаемый
случай, см. Q2).

## Взаимодействие с существующими фичами

F5 пересекается с четырьмя существующими контрактами. Каждое
пересечение фиксируется здесь, чтобы Spec не решал его неявно:

### F1 Property JSON (AC9)

F1 §6 фиксирует набор ключей Property JSON как стабильный. Добавление
`branch_id` — формальное изменение контракта. Это **аналог проблемы
F3 AC14** (F3 решил не трогать F2 Unit JSON, отдав amenities отдельным
subresource).

**Отличие F5 от F3:** здесь нет «отдельного subresource» как
альтернативы. `branch_id` — скалярное поле на Property, а не
коллекция. Единственная осмысленная форма — **включить в Property
JSON**. Это значит, что F5 **меняет** F1 контракт — не аддитивно,
а через breaking change на уровне ожидаемых ключей.

Spec должен решить, как это проговорить:

- Обновить F1 AC9 **в F1 spec** (ретроспективная правка чужого
  документа).
- Либо зафиксировать изменение **только в F5 spec**, явно сказав
  «F5 расширяет контракт F1 §6 полем `branch_id` — это формальное
  breaking change, но совместимое с клиентами, которые игнорируют
  незнакомые ключи».
- Либо обновить F1 тесты, расширив ожидаемый массив ключей на
  `branch_id`.

Рекомендация Spec-сессии: первый или третий вариант.

### F1 PropertiesController permit + security

F1 controller не знает про `branch_id`. Spec F5 должен явно указать:

- `branch_id` добавляется в `permit` списка.
- Резолв `branch_id` через scope (`Current.organization.branches.find_by`)
  — обязателен для security инварианта.
- **F4 lesson**: `belongs_to :branch, optional: true` silently
  resolves id глобально. F5 должен использовать тот же pattern —
  pre-save guard с `:not_in_scope` sentinel и direct 422 render в
  контроллере. **Без этого cross-org security invariant будет
  нарушен** (повторение F4 security bug).

### F1 Property model

F1 model не знает про Branch. F5 добавляет:

- `belongs_to :branch, optional: true`
- Опционально: custom validation `branch_must_exist_in_org`
  (аналог F4 `parent_branch_must_exist_in_org`), если Spec решит
  дублировать F4 defense-in-depth.

Это единственное изменение в существующих моделях F5.

### F4 DELETE /branches/:id (Q1)

F4 Spec §3.5.5 фиксирует restrict на удаление Branch с детьми
(`before_destroy` + `ON DELETE RESTRICT` на `parent_branch_id`). F5
вводит **второй** тип связи к Branch — через `Property.branch_id`.
Что происходит при попытке `DELETE /branches/:id`, если к этому
branch привязаны Property?

Варианты — вопрос Q1.

## Критический инвариант безопасности F5

**Property одной организации никогда не ссылается на Branch другой
организации.** Невозможно создать или отредактировать `Property`
так, чтобы `branch_id` указывал на Branch чужой организации.

Реализация должна **явно** повторить F4 pattern (прямой урок F4
implementation): pre-save scope resolve через
`Current.organization.branches.find_by(id: ...)`, на `nil` — **не**
полагаться на `belongs_to :branch, optional: true` (которая молча
резолвит `branch_id` глобально через `Branch.find`, крестя границы
организаций), а возвращать 422 прямо из контроллера через тот же
`:not_in_scope` sentinel pattern, что в F4
`BranchesController#resolve_parent_or_error`.

Это блокирующий security инвариант — нарушение = security bug,
квалифицируется как block-merge, а не как регрессия фичи. Spec
обязан явно покрыть инвариант тестами для POST и PATCH, независимо
от других AC.

## Что НЕ входит (фиксируется явно)

- **Endpoint `GET /api/v1/branches/:id/properties`** (список объектов
  филиала) — Phase 2.2.5, не HW-1.
- **Фильтрация `GET /api/v1/properties?branch_id=N`** — Phase 2.2.5.
- **Bulk перемещение** (все объекты одного филиала → другой) — вне
  scope.
- **Аналитика/агрегация по филиалам** — Phase 6.
- **Scope membership по branches** (региональный менеджер видит
  только свои филиалы) — Phase 1.3/F7.
- **Unit ↔ Branch direct link.** Unit наследует branch **транзитивно**
  через Property: `unit.property.branch_id`. F5 не вводит
  `Unit.branch_id` как отдельное поле и не добавляет helper-методов
  вроде `unit.branch`. Клиент или будущая аналитика Phase 6 строит
  связь через SQL join (`units JOIN properties ON ... JOIN branches`)
  или ORM цепочку. Транзитивность достаточна на HW-1 объёмах.
- **История перемещений** (audit log).
- **Миграция данных существующих Property.** Существующие Property
  получают `branch_id: null` — менеджер должен вручную распределить
  их по филиалам (не блокирующая операция).
- **Frontend.**
- Любые изменения моделей Unit, Amenity, UnitAmenity, Organization,
  Role, Membership, кроме добавления `has_many :properties` на
  `Branch` (§3.3-esque).

## Открытые вопросы (для Spec-сессии)

**Q1. Поведение `DELETE /api/v1/branches/:id` при наличии привязанных
Property — ON DELETE для `Property.branch_id`.**

Три варианта (аналог F4 Q3):

- **(а) Restrict** — запретить удаление Branch, у которого есть
  привязанные Property → 409 Conflict, аналог F3 Amenity в
  использовании. Менеджер должен сначала переместить или отвязать
  все объекты.
- **(б) Nullify** — при удалении Branch все его `Property.branch_id`
  становятся `null`. Property остаются в системе, просто отвязанные.
  Не теряет данных, но менеджер может удалить филиал «случайно» и
  потерять структурную информацию.
- **(в) Cascade** — удалить все Property вместе с Branch, и через
  cascade на Property → все Unit, Amenity-привязки и т.д. **Очень
  опасно** — одна команда удаляет всё дерево данных.

**Взаимодействие с F4 §3.5.5:** F4 уже реализовал restrict на
удаление Branch с **дочерними branches**. Q1 — отдельный случай
«Branch с привязанными Property». Варианты:

- Согласовано с F4: **restrict тоже** для Property — 409, если
  есть дети **или** есть привязанные property. Консистентно.
- Рассогласованно: разные правила для children vs properties.

Рекомендация Spec: **restrict** (а), консистентно с F4 и F3. F4
`before_destroy` callback можно расширить проверкой обоих условий,
или добавить отдельный callback.

**Отдельно от Q1 — DB-level поведение** (`on_delete` на FK
`properties.branch_id`). Q1 описывает ORM-level (`before_destroy`
с 409 ответом). DB-level — вторая линия защиты против прямых SQL DELETE,
решается в Q7 как отдельный вопрос. Оба уровня должны быть
согласованы, но они — разные решения. Вариант «только DB-level
без ORM callback» явно **не** рассматривается — он даёт 500
`InvalidForeignKey` вместо семантического 409 и не раскрывает
пользователю причину.

**Q2. `branch_id` в F1 Property JSON — ownership изменения контракта.**

Тесты F1 properties_spec.rb в любом случае нужно обновить (они
проверяют точный набор ключей через `match_array` — упадут без
правки). Это техническая необходимость, не вопрос. **Вопрос —
кому принадлежит изменение контракта:**

- **(а) F5 ретроспективно обновляет F1 Spec** (§6 JSON + AC9).
  Аргумент: чистый контракт, любой читатель F1 Spec видит актуальное
  состояние. Недостаток: F1 Spec status=`active` теряет
  immutability — каждая cross-cutting фича перерассматривает старые
  spec'и, «active» становится менее meaningful.
- **(б) F5 Spec owns изменение**, F1 Spec остаётся как был, в F5
  Spec явная секция «Changes to F1 contract». Аргумент: F1 Spec —
  snapshot во времени, изменения накапливаются в новых spec'ах.
  Недостаток: читатель F1 получает устаревший контракт, должен
  знать всю цепочку F5 → F1.

Это архитектурное решение про SDD-процесс, не только про F5. Spec
должен выбрать одну из двух моделей и зафиксировать — следующие
cross-cutting фичи будут использовать тот же подход.

Рекомендация: **(а)** — чистота актуального состояния важнее
исторической immutability. F1 Spec и так корректируется через
`PLAN.md` markers и `SCHEMA.md`. Но Spec-сессия вольна выбрать (б).

**Q3. Permission для смены `branch_id` у Property — `properties.manage`
или `branches.manage`?**

Варианты:

- **(а)** `properties.manage` — смена branch — это «редактирование
  property», purview существующего permission.
- **(б)** `branches.manage` — привязка к branch — это «реорганизация
  структуры», purview F4 permission.
- **(в)** Оба — пользователь должен иметь и то, и другое.

Spec решит. **(а)** — минималистичный вариант, в Spec-сессии выбрать,
если нет веских аргументов за большую гранулярность.

**Q4. Custom validation `branch_must_exist_in_org` в модели — нужна
или достаточно controller-level guard?**

F4 модель имела `parent_branch_must_exist_in_org` как defense-in-depth
— контроллер уже резолвил через scope, но модель тоже проверяла. В F5
можно сделать так же (консистентно с F4), или ограничиться только
controller-level (проще).

Рекомендация: консистентно с F4 — добавить custom-валидацию.
defense-in-depth важен для security-критического поля.

**Q5 / Q6 — не открытые вопросы, а констатации (зависят от Q1).**

F5 **обязана** изменить F4 модель `Branch`:

- Добавить `has_many :properties` (без этого controller-level и
  model-level проверки не могут запрашивать `branch.properties`).
- Если Q1 = restrict (а), расширить `before_destroy` callback
  проверкой `properties.exists?` (дополнительно к
  `children.exists?`) с тем же сообщением «Branch has
  dependent properties and cannot be deleted» или единым
  «Branch has dependents and cannot be deleted». Точная
  формулировка — задача Spec.

Альтернативы (монкипатч, class_eval в F5, checking only in F1
controller) отвергнуты — нарушают Rails конвенции и не-Rails way.

**Это единственный случай в HW-1, когда одна фича меняет модель
другой.** F5 touches F4 model minimally (две строки или около).
Spec F5 обязан явно зафиксировать это как часть своего scope;
регресс-гард F4 тестов должен покрывать `before_destroy` с новой
логикой.

**Q7. Миграция — `add_reference :properties, :branch, foreign_key:
true, null: true`.**

Стандартный Rails pattern. Единственный вопрос — `on_delete`
behavior на DB-уровне:

- `on_delete: :restrict` — консистентно с F4 self-FK, защита от
  прямых SQL DELETE.
- `on_delete: :nullify` — если Q1 = (а) restrict на ORM-уровне, DB
  может делать nullify (разные уровни, как F3).
- Без `on_delete` — PostgreSQL default `NO ACTION` ≈ restrict.

Рекомендация: явно `on_delete: :restrict`, консистентно с F4. Либо
оставить default, но зафиксировать в Spec.

**Q8. Нужно ли F5 делать индекс на `branch_id`?**

Rails `add_reference :properties, :branch` по умолчанию создаёт
индекс. Это нужно для быстрого lookup «Branch has_many :properties»
(даже если endpoint вне scope F5, ассоциация `branch.properties`
внутри `before_destroy` callback требует эффективного querying).
Оставить default. Не open question, фиксирую.
