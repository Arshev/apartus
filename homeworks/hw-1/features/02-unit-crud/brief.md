# Brief — Unit CRUD

## Контекст

Apartus — PMS для управления арендной недвижимостью. На момент старта F2
в системе уже есть `Property` (F1, HW-1): менеджер может завести в
организации объекты недвижимости — дом, отель, апартаменты, хостел. Но
**«дом» сам по себе не сдаётся**. Сдаётся физический юнит внутри него:
конкретная квартира, номер в отеле, комната в хостеле, сам дом как
целое — в зависимости от типа Property. Без сущности Unit у Apartus нет
объекта, на который:

- вешается цена (Phase 4: Rates, SeasonalPrice, LengthDiscount);
- ставится бронирование (Phase 3: Reservation принадлежит Unit, не Property);
- привязываются удобства (F3 HW-1: Amenity через UnitAmenity);
- считается загрузка / занятость (Phase 6 аналитика: occupancy, RevPAR, ADR);
- формируется наряд-заказ и инспекция (Phase 8).

Unit в доменной модели существует только внутри конкретного Property:
юнит без родительского объекта недвижимости не имеет смысла и не
рассматривается вне связи с ним. Это инвариант предметной области, а
не деталь реализации.

У юнитов собственный классификатор типов, не совпадающий с классификатором
типов Property. Тип Property описывает здание целиком (квартира, отель,
дом, хостел), тип юнита описывает физическую единицу сдачи внутри него
(например, номер, комната, кровать, студия) — смешение этих двух
классификаторов семантически некорректно.

F2 — первая фича после reference implementation F1 и первая точка,
в которой процесс Brief → Spec → Plan → Implement проживается «по
шаблону», сформированному на F1.

## Проблема

Менеджер арендного бизнеса, который завёл в Apartus Property своей
организации, **не может указать, что именно внутри этого объекта сдаётся
в аренду**. Из-за этого:

- Property остаётся «пустой коробкой»: в отеле не перечислены номера, в
  доме под разделение — комнаты, в апарт-комплексе — квартиры. Менеджер
  не видит, сколько у него вообще сдаваемых единиц и какие они.
- Невозможно продолжать разработку всех модулей, которые работают с
  физической единицей сдачи: бронирование, цены, amenities, загрузка,
  наряд-заказы. Все они опираются на Unit как на первичный ключ
  предметной области и без него не имеют смысла.
- Онбординг останавливается на шаге после создания Property:
  «добавил объект — а дальше ничего не могу сделать». Продукт
  функционально не продвигается от «завёл здание» к «готов принимать
  гостей».

## Стейкхолдеры

- **Первичный:** менеджер арендного бизнеса (роль `manager` или `owner`
  в Membership) — заводит юниты внутри объектов своей организации,
  указывает их тип, вместимость и текущий статус, редактирует по мере
  изменений (ремонт, блокировка, возврат в работу).
- **Вторичный:** разработчик Apartus — получает первую проверку
  reference-паттерна F1 в работе на следующей фиче. Наблюдения фиксируются
  в `homeworks/hw-1/report.md` и влияют на то, как будут делаться F3–F5.

## Желаемый результат

После выполнения фичи менеджер с правом управления юнитами (роль
`manager` или `owner` в Membership, либо иная роль с соответствующим
разрешением), авторизованный в своей организации, может для любого
Property этой организации:

1. Посмотреть список всех юнитов выбранного Property.
2. Открыть детали конкретного юнита.
3. Завести новый юнит, указав атрибуты: название, тип юнита,
   вместимость, текущий статус.
4. Изменить атрибуты существующего юнита.
5. Удалить юнит, который больше не нужен.

После создания юнит закреплён за своим Property и **не переносится
в другой Property** — ни через API, ни иным способом. «Перевешивание»
юнита на другой родительский объект не поддерживается: если такая
операция понадобится в реальной жизни (перенумерация, переоборудование),
она моделируется как удаление старого юнита и создание нового.

Юниты одного Property **никогда не видны и не доступны** при запросе
против другого Property — даже внутри той же организации, даже если
у пользователя есть полные права на оба объекта. Такая попытка не
раскрывает факт существования юнита.

Юниты организации A **никогда не видны и не доступны** пользователям
организации B — ни на чтение, ни на запись. Это инвариант уровня
организации, унаследованный от F1.

Пользователь с ограниченной ролью (`units.view`, без `units.manage`)
может только читать список и открывать детали, но не создавать,
изменять или удалять юниты. Пользователь без `units.view` не видит
юниты вообще.

## Критерии успеха (что считаем «фича работает»)

- Менеджер проходит полный цикл «создал юнит внутри Property → нашёл в
  списке этого Property → открыл → изменил → удалил» без ошибок на
  валидных входных данных.
- Попытка получить доступ к юниту через другой Property (внутри своей
  организации) или через Property другой организации не раскрывает
  факт существования ни юнита, ни Property.
- Пользователь с правом только на чтение юнитов может читать, но
  получает отказ при попытке создавать, изменять или удалять.
- Поведение фичи зафиксировано автоматическими тестами; регрессия
  блокируется CI и порогом coverage ratchet.
- Модель Unit готова служить целевой сущностью для последующих фич
  HW-1 (Amenities — F3) и фаз проекта (Reservations — Phase 3, Rates —
  Phase 4).

## Ограничения и явные НЕ-цели

**НЕ входит в F2:**

- Amenity / UnitAmenity — фича F3. F2 **не** вводит понятие удобств
  и не резервирует под них поля.
- Загрузка фото юнита (Active Storage) — отложено на Phase 2.1.4.
- Статусный workflow: правила переходов между статусами юнита и
  side-effects при смене статуса. F2 позволяет менеджеру менять статус,
  но не накладывает ограничений на то, какие переходы допустимы.
- Bulk-операции (массовое создание N юнитов сразу, копирование из
  другого Property) — вне scope.
- Поиск, сортировка, фильтрация за пределами простого `index` — вне
  scope.
- Soft delete не используется — Unit удаляется физически.
- Назначение ответственных сотрудников (housekeeper, inspector,
  технический специалист) на юнит — Phase 8 (Tasks & Maintenance).
- История изменений юнита (audit log: кто и когда менял атрибуты или
  статус) — в F2 не ведётся.
- Frontend (Vue-страницы, формы, списки) — вне HW-1 backend scope.

**Технические ограничения (из CLAUDE.md и F1, обязательны, не обсуждаются):**

- Строгая изоляция данных между организациями — инвариант F1, F2 его
  наследует: юниты одной организации недоступны пользователям другой
  ни при каких условиях.
- Аутентификация и авторизация поверх существующих механизмов проекта
  (Rails 8 built-in auth, Pundit, `Permissions` concern) — не вводить
  параллельных.
- Никаких новых gems/npm-пакетов без явного разрешения.
- Существующие миграции не изменяются — только новые.

## Почему сейчас

1. F2 — вторая фича HW-1 и логический шаг Phase 2 в `ai-docs/PLAN.md`
   (пункты 2.1.2 и 2.2.2). Без Unit все последующие модули проекта
   (Reservation, Rate, Amenity, WorkOrder) не имеют базовой сущности.
2. F2 — **первая проверка reference implementation F1** на
   переиспользуемость. Откладывать эту проверку опасно: если эталон
   плохо ложится на фичу, где сущность существует только внутри
   родителя, узнать об этом лучше на F2, чем обнаружить несовместимость
   на F3 или F5, когда по эталону уже сделано три фичи.
3. Сводная сдача HW-1 требует не только эталона, но и демонстрации
   **масштабирования** процесса Brief → Spec → Plan → Implement на
   нескольких фичах подряд. F2 — первая фича, которую процесс
   должен прожить «по шаблону».
